# Metamorphic Eval

## Project Structure

```
thesis-metamorphic-eval/
├── agents/
├── benchmarks/
    ├── downloads/
├── code-coccoon/
├── repos/
├── results/
├── scripts/
    ├── transform.py
    ├── bootstrap.py
├── swe_bench/
├── transformations/
```

### Purpose

1. `agents`: The [bootstrap.py](./scripts/bootstrap.py)'s `init` subcommand downloads the [MSWE-agent](https://github.com/Vladislav0Art/MSWE-agent) into this folder.
1. `benchmarks`:
    1. `./downloads`: Multi-SWE-bench and Multi-SWE-bench_mini are downloaded here by the [bootstrap.py](./scripts/bootstrap.py)'s `benchmark` subcommand.
    1. The folder itself is intended to contain JSONL files with the benchmark definitions (see example in [sample.jsonl](./benchmarks/sample.jsonl))
1. `code-coccoon`: The [bootstrap.py](./scripts/bootstrap.py)'s `init` subcommand downloads the [CodeCocoon-Plugin](https://github.com/JetBrains-Research/CodeCocoon-Plugin) into this folder.
1. `repos`: The [transform.py](./scripts/transform.py) script will download repositories from the benchmarks JSONL file into this folder and run CodeCocoon's transformations on them.
1. `results`: Thesis results are intended to be stored here.
1. `scripts`: scripts are store in this folder ([transform.py](./scripts/transform.py), [bootstrap.py](./scripts/bootstrap.py))
1. `swe_bench`: The [bootstrap.py](./scripts/bootstrap.py)'s `init` subcommand downloads the [multi-swe-bench](https://github.com/Vladislav0Art/multi-swe-bench) into this folder.
1. `transformations`: JSON files with transformations' definitions for the [codecoccoon.yml](https://github.com/JetBrains-Research/CodeCocoon-Plugin/blob/main/codecocoon.example.yml) files generated by [transform.py](./scripts/transform.py) are store in this folder (see [sample.json](./transformations/sample.json)).

## Getting Started

### Workflow

1. **Preparation**:
    1. Initialize repositories (_CodeCocoon-Plugin, MSWE-agent, multi-swe-bench_) (*)
    1. Install the needed (sub)set of benchmarks as a JSONL file (_**Multi-SWE-Bench** or **Multi-SWE-Bench-mini** supported_).
    1. Generate CodeCocoon's metamorphic transformations and insert their diffs into the benchmarks JSONL file.
1. **Execution**:
    1. Execute MSWE-agent on both versions (non-metamorphic and metamorphic) of the benchmarks (_the matter of applied patches to the base commit of each benchmark_).
1. **Evaluation**:
    1. Run **multi_swe_bench** repo's evaluation script to get final eval metric.
    1. [**Additionally**] Given the agent's output on non-metamorphic and metamorphic diffs, calculate some other metrics (_not yet implemented_).

### Preparation

The `bootstrap.py` script sets up the evaluation environment by downloading all 3 repos (*) and collecting a set of benchmarks as a single JSONL file.

#### 1. Initialization

Downloads required repositories:

```bash
python scripts/bootstrap.py init
# Or with custom URLs:
python scripts/bootstrap.py init --codecoccoon <url> --mswe_agent <url> --multi_swe_bench <url>
```

Run the help command to see the default URLs:
```bash
python scripts/bootstrap.py init --help
```


#### 2. Benchmarks

Creates filtered benchmark datasets:
```bash
# Basic usage
python scripts/bootstrap.py benchmark Multi-SWE-bench_mini --language java --difficulty easy

# With random sampling (selects 10 random out of all JSONL entries)
python scripts/bootstrap.py benchmark Multi-SWE-bench --language java --difficulty medium --rand 10

# With custom output path
python scripts/bootstrap.py benchmark Multi-SWE-bench_mini --language python --output my-benchmark.jsonl
```

Run the help command to see the supported values of every parameter:

```bash
python scripts/bootstrap.py benchmark --help
```

#### 3. Metamorphic Transformations

The [transform.py](./scripts/transform.py) executes the following algorithm:

1. Accepts the set of benchmarks collected by the [bootstrap.py](./scripts/bootstrap.py)'s `benchmark` subcommand.
1. For every benchmark, clones its repo into `repos/{benchmark.instance_id}` and runs [CodeCocoon-Plugin](https://github.com/JetBrains-Research/CodeCocoon-Plugin) that applies transformations from the JSON file passed as a filepath via `-t/--transformations` parameter (see [sample.json](./transformations/sample.json)).
1. The diffs after applying metamorphic transformations will be saved into every benchmark's JSON entry and saved as a new JSONL file under a filepath passed as `-o/--output` parameter.

The schema is:

```json
{
    // ... other benchmark entries ...
    "base": {
        // other fields copied from the input entry
        "metamorphic_base_patch": "same as `metamorphic_base_patch` below"
    },
    "test_patch": "save as `new_morphed_test_patch`",
    "metamorphic": {
        "[strategy-name]": {
            "metamorphic_base_patch": "diff --git ...",
            "metamorphic_base_commit": "18a57a3299d1db5338c08991a511b0c4d36a7b2a",
            "logs": [
                {
                    "applied_to": "base",
                    "label": "base_metamorphic_transformation_log",
                    "result": {
                        "stdout": "...",
                        "stderr": "...",
                        "return_code": 0
                    }
                },
                {
                    "applied_to": "test",
                    "label": "test_metamorphic_transformation_log",
                    "result": {
                        "stdout": "...",
                        "stderr": "...",
                        "return_code": 0
                    }
                }
            ],
            "_metamorphic_test_patch": "diff --git ...",
            "metamorphic_test_commit": "1a60af4a8d773bb3e3a0caba938da72545a3f91f",
            "original_test_patch": "diff --git ...",
            "new_morphed_test_patch": "diff --git ..."
        }
    }
}
```

Where `strategy-name` is passed as a `-s/--strategy` parameter of the [transform.py](./scripts/transform.py) script.


**IMPORTANT**:

The idea is to morph the input benchmarks and procude a fully new dataset with metamorphic transformations applied, so that MSWE-agent and multi_swe_bench eval can treat the morphed benchmarks in _almost_ the same way they do with unmodified ones (_NOTE: "almost" because the metamorphic patch applied on the base commit should be explicitly applied on the base commit again during image build by both MSWE-agent and multi_swe_bench eval_).

1. `test_patch` entry will be replaced with the new `new_morphed_test_patch` patch, which is a diff between `(base + metamorphic_base_patch)` and `(base + test_patch (original) + _metamorphic_test_patch)`. This is implemented this way to minimize modifications in MSWE-agent and multi_swe_bench repos. The original `test_patch` present in the input is stored under `metamorphic.original_test_patch`.
1. `base.metamorphic_base_patch` is a metamorphic modification produced by CodeCocoon-Plugin on the `base` commit. This patch is expected to be applied when present by the MSWE-agent and multi_swe_bench during image build to build a new base commit as the original base + metamorphic transformation on top of it.


Run the help command to see the list of available parameters and their purpose:
```
python scripts/transform.py --help
```

**General usage**:

Given filepaths will be converted to absolute filepaths.

Here is a runnable example:

```bash
python scripts/transform.py -i ./benchmarks/sample.jsonl \
    -o ./benchmarks/metamorphic-sample.jsonl \
    -s 'MyStrategy' \
    -c ./code-coccoon/CodeCocoon-Plugin \
    -r ./repos \
    -e /path/to/env/file \ # filepath to ENV file that will provided to CodeCocoon (optional)
    -t ./transformations/sample.json \
    --override
```

CodeCocoon needs `GRAZIE_TOKEN` for some transformations. To supply additional ENV variables, create an `.env` file (any name allowed) with a key-value entries, and pass it as `-e/--env_filepath` parameter. CodeCocoon subprocess will be executed with ENV variables of the parent process along with the ENV variables defined in the provided `.env` file:
```bash
GRAZIE_TOKEN="your-grazie-token"
```

After running, navigate to `benchmarks/metamorphic-sample.jsonl` and search for `"metamorphic"` JSON field.

### Execution

Executing MSWE-agent on the normal and metamorphically changed benchmarks.

#### Prerequisites

1. Change directory to `agents/MSWE-agent`:
```bash
cd agents/MSWE-agent
```

1. Create `keys.cfg` file in the root directory of the **MSWE-agent** repo with the following content:
```
OPENAI_API_KEY="[your-OpenAI-key]"
```

1. Switch a branch to `vartiukhov/metamorphic-testing`:
```bash
git switch vartiukhov/metamorphic-testing
```

1. MSWE-agent has its own python project run via **uv**; initialize the project by running:
```bash
uv venv
uv pip install -r requirements.txt
```

1. Activate the created venv:
```bash
. .venv/bin/activate
```

1. Run Docker and see whether it's accessible from CLI:
```bash
docker --version
```

#### MSWE-agent Keys

These are the custom keys for `keys.cfg` (added in the MSWE-agent's branch `vartiukhov/metamorphic-testing`):

1. **LiteLLM Support**:
   1. `USE_LITELLM_PROXY=True` - whether to call LiteLLM.
   1. `LITELLM_OPENAI_BASE_URL=url` - LiteLLM base url for OpenAI model.
   1. `LITELLM_API_KEY` - LiteLLM API key.
1. **Langfuse Support**:
   1. `TRACE_TO_LANGFUSE=True` - whether to enable Langfuse tracing.
   1. `LANGFUSE_HOST=url` - Langfuse host.
   1. `LANGFUSE_SECRET_KEY=skey` - secret key.
   1. `LANGFUSE_PUBLIC_KEY=pkey` - public key (_a pair of both keys defines the project which the traces will be exported to_).
1. **Grazie Support**:
   1. `USE_GRAZIE_PROXY=True` - whether to call Grazie (**call LiteLLM instead!**)
   1. `GRAZIE_OPENAI_BASE_URL=url` - Grazie base url for OpenAI model.
   1. `GRAZIE_ANTHROPIC_BASE_URL=url` - Grazie base url for Anthropic model
   1. `GRAZIE_API_KEY=key` - Grazie API key.


#### MSWE-agent On Normal Benchmarks

1. Run the MSWE-agent from the `agents/MSWE-agent` with the following command (running on the sample benchmark [sample.jsonl](./benchmarks/sample.jsonl)):
```bash
python run.py \
   --model_name gpt4o \
   --cache_task_images True \
   --pre_build_all_images True \
   --remove_image False \
   --pr_file /path/to/thesis-metamorphic-eval/benchmarks/sample.jsonl \
   --config_file /path/to/thesis-metamorphic-eval/agents/MSWE-agent/config/default.yaml \
   --skip_existing=True \
   --per_instance_cost_limit 5.00 \
   --print_config=False \
   --max_workers_build_image 16
```

Once finished, the model predications will be saved somewhere at the [agents/MSWE-agent/trajectories](agents/MSWE-agent/trajectories) folder:
```
trajectories/user/model__sample__default__t-0.00__p-0.95__c-5.00__install-1/all_preds.jsonl
```

1. Finally, convert this JSONL file to the format expected by the multi_swe_bench evaluation script:

Navigate into the [scripts](./scripts/) folder and run the [scripts/convert_model_predictions.py](./scripts/convert_model_predictions.py):
```
python convert_model_predictions.py \
    -i /path/to/thesis-metamorphic-eval/agents/MSWE-agent/trajectories/.../all_preds.jsonl \
    -o /path/to/thesis-metamorphic-eval/results/predictions/fix_patches.jsonl
```

The resulting `fix_patches.jsonl` file will be supplied into multi_swe_bench evaluation script to generate final reports.



#### MSWE-agent On Metamorphically Changed Benchmarks

The difference will be in the input JSONL file given to the MSWE-agent. Once you morph the initial JSONL with benchmarks via [transform.py](./scripts/transform.py) script, you can pass it directly to the MSWE-agent as a normal input: The updated implementation in the `vartiukhov/metamorphic-testing` branch will **apply and commit** the `base.metamorphic_base_patch` patch during benchmark image build to treat this patched base as a new base commit. Since [transform.py](./scripts/transform.py) script replaces `test_patch` with the morphed test patch, it will be used in the evaluation same as a normal `test_patch`.

At this point you are expected to have a JSONL file with model patches in the [results/predictions](./results/predictions) folder.
Assuming there is a `fix_patches.jsonl` file, which contains the following JSON entries:

{

#### Multi-SWE-Bench: Normal Evaluation

1. Set up config file for the evaluation:

The evaluation script accepts a `config.json` file, an example of which you can find in [swe_bench/config.example.json](./swe_bench/config.example.json).

You need to set the following parameters:
1. `workdir`: set to `swe_bench/artifacts/workdir`
1. `patch_files`: set to the generated `results/predictions/fix_patches.jsonl`.
1. `dataset_files`: set to the benchmark file, on which the MSWE-agent was running. **IMPORTANT: The filename MUST contain the language word (in our case, 'java')!**
1. `output_dir`: set to `results/predictions/output`
1. `repo_dir`: set to `swe_bench/artifacts/repos`
1. `log_dir`: set to `swe_bench/artifacts/logs`

Read about the config parameters [here](https://github.com/Vladislav0Art/multi-swe-bench).


1. Next, from the [swe_bench/multi-swe-bench](./swe_bench/multi-swe-bench/) directory, run the evaluation pipeline pointing to the created config:
```bash
python -m multi_swe_bench.harness.run_evaluation \
    --config /path/to/thesis-metamorphic-eval/swe_bench/config.example.json
```

After the execution you will see the following output:

1. `swe_bench/artifacts/workdir`:
    - Contains directories with instances' patches and their reports (in `report.json` file), along with the log.

1. `results/predictions/output/final_report.json`:
    - Contains information about how many of the patch instances resolved correctly. The **final PASS metric** is calculated as `resolved_instances / total_instances * 100%`.

1. `swe_bench/artifacts/repos`:
    - Benchmark repositories are cloned into this folder; next, the model's fix patch is applied and the repo tests are run.

1. `swe_bench/artifacts/logs`:
    - Evaluation (`run_evaluation.log`) and report (`gen_report.log`) logs are written into this folder.


From the `final_report.json` you may derive some metrics.

### Evaluation Metrics

When running the pipeline via [scripts/evaluate.py](./scripts/evaluate.py), two metric artifacts are
produced automatically after each completed run.

#### Per-run metrics (`run-i/result.json`)

**Agent execution metrics** — `agent.metrics`

After each run the agent step reads every `.traj` file it produced and extracts
`info.model_stats` from it:

```
agent.metrics.execution   — array, one entry per instance
agent.metrics.summary     — avg / median over all instances in this run
```

Each execution entry:
```json
{
  "instance_id":     "mockito__mockito-3129",
  "total_cost":      5.07,
  "instance_cost":   5.07,
  "tokens_sent":     1002772,
  "tokens_received": 3666,
  "api_calls":       61
}
```

The per-run summary:
```json
{
  "n_instances": 11,
  "n_missing":   0,
  "total_cost":  {"avg": 3.14, "median": 2.88},
  "tokens_sent": {"avg": 500000, "median": 450000},
  "tokens_received": {"avg": 3500, "median": 3200},
  "api_calls":   {"avg": 55.0, "median": 50.0}
}
```

`n_missing` counts instances whose `.traj` file was absent or malformed;
those are excluded from the numeric aggregations.

**Evaluation (pass rate) metrics** — `evaluation.metrics`

After the harness finishes, `final_report.json` is read and the three
primary fields are extracted along with the computed pass rate:

```json
"evaluation": {
  "metrics": {
    "summary": {
      "total_instances":      11,
      "resolved_instances":   5,
      "unresolved_instances": 6,
      "pass_rate":            45.5
    }
  }
}
```

`pass_rate = resolved / total × 100` (expressed as a percentage 0–100).


#### Cross-run metrics (`workdir/metrics_summary.json`)

After all N runs complete, the orchestrator aggregates all per-run data
into a single file at `{workdir}/metrics_summary.json`.

**`pass_rate`** — the primary benchmark metric

The pass rate for each run is averaged across N runs.  Because `total_instances`
is the same in every run (same benchmark), the average pass rate equals the
pooled pass rate, so no separate "pooled" computation is needed.

```json
"pass_rate": {
  "avg":  45.5,
  "std":   3.2,
  "min":  40.0,
  "max":  50.0,
  "per_run": [
    {"run_number": 1, "resolved": 5, "total": 11, "pass_rate": 50.0},
    {"run_number": 2, "resolved": 4, "total": 11, "pass_rate": 40.0}
  ]
}
```

`std` is the standard deviation of per-run pass rates.  Use it as the ±
value in thesis tables: *"pass rate was 45.5 ± 3.2 %"*.  `pass_rate` is only
present in the file when the evaluation step actually ran.

**`pooled`** — agent cost / token headline metric

All N × M per-instance cost/token observations are merged into one flat list
and avg / median are computed over the whole pool.  Answers: *"On average,
how much does a single agent invocation cost on this benchmark?"*

```json
"pooled": {
  "n_observations": 22,
  "total_cost":  {"avg": 3.14, "median": 2.88},
  "tokens_sent": {"avg": 500000, "median": 450000}
}
```

**`run_variability`** — agent cost / token stability metric

The per-run averages of cost/token fields are described with mean, std dev,
min, and max.  Answers: *"How consistent is the agent across runs?"*

```json
"run_variability": {
  "total_cost": {
    "avg_of_run_avgs": 3.14,
    "std_of_run_avgs": 0.02,
    "min_run_avg":     3.12,
    "max_run_avg":     3.16
  }
}
```

**`per_run`** — per-run breakdown (for notebook plotting)

Each entry holds both agent and evaluation summaries for that run:

```json
"per_run": [
  {
    "run_number": 1,
    "agent":      {"n_instances": 11, "total_cost": {"avg": 3.14, ...}, ...},
    "evaluation": {"total_instances": 11, "resolved_instances": 5,
                   "unresolved_instances": 6, "pass_rate": 50.0}
  }
]
```

`evaluation` is `null` for runs where the evaluation step did not execute.

**Summary of views**

| Key | Source | Question answered | Thesis use |
|---|---|---|---|
| `pass_rate` | `final_report.json` | Did the agent fix the bugs? | Primary benchmark metric |
| `pooled` | `.traj` model_stats | How much did it cost / how many tokens? | Headline cost metric |
| `run_variability` | Per-run averages | Is the agent consistent across runs? | Error bars, stability |
| `per_run` | Both | Per-run detail | Notebook plots, traceability |

The math functions are in [`scripts/eval/metrics.py`](./scripts/eval/metrics.py)
and can be imported directly into Jupyter notebooks without loading the rest
of the pipeline.
